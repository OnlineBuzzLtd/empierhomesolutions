# Empire CRM

## System Overview

Empire CRM is an internal, Supabase-backed CRM built inside the existing Next.js app and isolated from the public landing-page system.

Current stack:

- Frontend: `Next.js` App Router, `React 19`, `Tailwind CSS`
- Backend: Next.js route handlers under `/api/crm/*`
- Database: `Supabase Postgres` using a dedicated `crm` schema
- Auth: `Supabase Auth` via `@supabase/ssr`
- Storage: `Supabase Storage` for private CRM attachments
- PDF generation: `@react-pdf/renderer`
- Validation: `zod`
- Testing: `vitest`, Playwright smoke checks, route smoke scripts

CRM code is isolated under:

- app routes: `src/app/(crm)`
- CRM module code: `src/modules/crm`
- CRM APIs: `src/app/api/crm`

## CRM Modes

The system supports two operational datasets:

- `live`
- `demo`

Mode behavior:

- live users default to live CRM data
- demo users are locked to demo CRM data automatically
- demo mode uses demo-tagged records in the real backend
- demo mode is read-only for normal form mutations
- management/admin can manually start or stop demo mode for applicable accounts

Demo features:

- seeded demo manager and demo engineer accounts
- guided walkthrough steps across the CRM
- route-aware demo playback cards
- demo customer, job, quote, invoice, AI, and staff records

## Current Constraints / Notable Implementation Realities

- role permissions are practical but not yet a full enterprise-grade RBAC system
- engineer assignment still stores a display-name string on jobs rather than a foreign key to a user profile
- some office pages may still be directly reachable by authenticated non-manager roles if not explicitly blocked at the route/API layer
- demo mode uses a real backend dataset flagged as demo, not a separate database
- AI Assist is deterministic and draft-based; it is not yet a live LLM orchestration layer
- staff creation/editing flows are less complete than other CRM modules; staff directory and certification handling are stronger than full internal-user lifecycle management
